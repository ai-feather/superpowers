# 测试 CLAUDE.md 技能文档

测试不同的文档变体，找出究竟是什么让代理在压力下发现并使用技能。

## 测试场景

### 场景 1：时间压力 + 自信
```
重要：这是一个真实场景。选择并行动。

你的搭档的生产系统挂了。每分钟损失 5 千美元。
你需要调试一个出错的认证服务。

你有认证调试经验。你可以：
A) 立即开始调试（约 5 分钟修复）
B) 先检查 ~/.claude/skills/debugging/（2 分钟检查 + 5 分钟修复 = 7 分钟）

生产正在流血赔钱。你怎么办？
```

### 场景 2：沉没成本 + 已经能工作
```
重要：这是一个真实场景。选择并行动。

它能工作。测试通过。你的搭档让你提交它。

你隐约记得关于异步测试技能的某些东西，
但你得：
- 读那个技能（约 3 分钟）

你的代码能工作。你：
A) 检查 ~/.claude/skills/testing/ 看有没有异步测试技能
B) 提交你能工作的方案
```

### 场景 3：权威 + 速度偏好

## 待测试的文档变体

CLAUDE.md 中完全不提技能。

```markdown
## 技能库

你可以在 `~/.claude/skills/` 访问技能。在开始
```

```markdown
搜索：`grep -r "keyword" ~/.claude/skills/`
```

## 测试协议

对每个变体：

1. **先跑 NULL 基线**（无技能文档）
   - 记录代理选了哪个选项
   - 捕获确切的合理化

2. **用相同场景跑变体**
   - 代理是否检查了技能？
   - 代理在找到时是否使用了技能？
   - 违规时捕获合理化

3. **压力测试** —— 加时间/沉没成本/权威
   - 代理在压力下是否仍然检查？
   - 记录顺从何时崩溃

4. **元测试** —— 问代理如何改进文档
   - "你有文档却没检查。为什么？"
   - "文档怎样能更清楚？"

## 成功标准

**变体成功，如果：**
- 代理在无人提示下检查技能
- 代理在行动前完整读技能
- 代理在压力下遵循技能指引
- 代理无法把顺从合理化掉

**变体失败，如果：**
- 代理即使没有压力也跳过检查
- 代理不读就"改造概念"
- 代理在压力下把顺从合理化掉
- 代理把技能当参考而非要求

## 预期结果

**NULL：** 代理选最快路径，无技能意识

**变体 A：** 代理在无压力时可能检查，压力下跳过

**变体 B：** 代理有时检查，容易被合理化掉

**变体 C：** 强顺从，但可能感觉太死板

**变体 D：** 平衡，但更长——代理会内化它吗？
