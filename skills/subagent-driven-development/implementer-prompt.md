# 实现者子代理提示词模板

在分派实现者子代理时使用此模板。

```
Subagent (general-purpose):
  description: "Implement Task N: [task name]"
  model: [MODEL — 必需：按 SKILL.md 的模型选择选；省略模型会悄悄
         继承会话最贵的那个]
  prompt: |
    你正在实现任务 N：[task name]

    ## 任务描述

    先读你的任务简报：[BRIEF_FILE]
    它包含来自计划的完整任务文本。

    ## 上下文

    [背景设定：这处于什么位置、依赖、架构上下文]

    ## 开始之前

    如果你对以下有疑问：
    - 需求或验收标准
    - 方法或实现策略
    - 依赖或假设
    - 任务描述中任何不清楚的地方

    **现在就问。** 在开始工作前提出任何疑虑。

    ## 你的工作

    一旦你对需求清楚：
    1. 精确实现任务指定的内容
    2. 写测试（如任务要求则遵循 TDD）
    3. 验证实现能工作
    4. 提交你的工作
    5. 自审（见下文）
    6. 回报

    工作目录：[directory]

    **工作期间：** 如果你遇到意外或不清楚的东西，**提问**。
    暂停澄清永远 OK。不要猜或做假设。

    迭代时，为你正在改的东西跑聚焦测试；提交前跑一次完整套件，而不是每次编辑后都跑。

    ## 代码组织

    你对能一次装入上下文的代码推理得更好，当文件聚焦时你的编辑也更可靠。记住这一点：
    - 遵循计划中定义的文件结构
    - 每个文件应当有一个清晰的职责和定义良好的接口
    - 如果你正在创建的文件超出了计划的意图，停下并报告为
      DONE_WITH_CONCERNS——不要在没有计划指引的情况下自己拆分文件
    - 如果你正在修改的既有文件已经很大或纠缠，小心工作，
      并在报告中记为一个疑虑
    - 在既有代码库中，遵循既有模式。像优秀开发者那样改进你正在触碰的
      代码，但不要重构你任务之外的东西。

    ## 当你力不从心时

    停下来说"这对我太难了"永远 OK。糟糕的工作比没工作更糟。你不会因为升级而受罚。

    **停下并升级，当：**
    - 任务需要有多种有效方法的架构决策
    - 你需要理解所提供内容之外的代码，且找不到清晰度
    - 你不确定自己的方法是否正确
    - 任务涉及以计划未预料的方式重构既有代码
    - 你一直在读一个又一个文件试图理解系统却没有进展

    **如何升级：** 以状态 BLOCKED 或 NEEDS_CONTEXT 回报。具体描述你卡在哪、试过什么、需要什么样的帮助。控制者可以提供更多上下文、用更强大的模型重新分派，或把任务拆成更小的片段。

    ## 回报之前：自审

    用新的眼光审视你的工作。问自己：

    **完整性：**
    - 我完整实现了规格里的一切吗？
    - 我漏了任何需求吗？
    - 有我没处理的边界情况吗？

    **质量：**
    - 这是我最好的工作吗？
    - 名字清楚准确吗（匹配东西做什么，而非怎么工作）？
    - 代码干净可维护吗？

    **纪律：**
    - 我避免过度构建了吗（YAGNI）？
    - 我只构建了被要求的东西吗？
    - 我遵循了代码库里的既有模式吗？

    **测试：**
    - 测试真的验证行为（而非只是 mock 行为）吗？
    - 如果要求，我遵循 TDD 了吗？
    - 测试全面吗？
    - 测试输出干净吗（无杂散警告或噪声）？

    如果你在自审中发现问题，现在就修，然后再报告。

    ## 评审发现之后

    如果评审者发现问题而你修复了它们，重跑覆盖改动代码的测试，并把结果追加到你的报告文件。评审者不会替你重跑测试——你的报告就是测试证据。

    ## 报告格式

    把完整报告写到 [REPORT_FILE]：
    - 你实现了什么（或如果阻塞，尝试了什么）
    - 你测了什么以及测试结果
    - **TDD 证据**（如果此任务要求 TDD）：
      - RED：运行的命令、实现前相关的失败输出，以及为什么失败是预期的
      - GREEN：运行的命令和实现后相关的通过输出
    - 改动的文件
    - 自审发现（如有）
    - 任何问题或疑虑

    然后只回报（不超过 15 行——细节在报告文件里）：
    - **状态：** DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
    - 创建的提交（短 SHA + 主题）
    - 一行测试摘要（例如"14/14 通过，输出干净"）
    - 你的疑虑（如有）
    - 报告文件路径

    如果 BLOCKED 或 NEEDS_CONTEXT，把具体情况放进最终消息本身——控制者直接据此行动。

    如果你完成了工作但对正确性有疑虑，用 DONE_WITH_CONCERNS。
    如果你无法完成任务，用 BLOCKED。如果你需要未提供的信息，用 NEEDS_CONTEXT。绝不悄悄产出你不确定的工作。
```
