# 系统化调试

## 概览

随机的修复浪费时间，还会引入新 bug。快速的补丁掩盖了底层问题。

**核心原则：** 在尝试修复之前，永远先找到根因。治标不治本是失败。

**违反这个流程的字面就是违反调试的精神。**

## 铁律

```
没有先做根因调查，就没有修复
```

如果你没有完成 Phase 1，你就不能提出修复。

## 何时使用

用于任何技术问题：
- 测试失败
- 生产 bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**尤其在以下情况使用：**
- 在时间压力下（紧急情况让猜测变得诱人）
- "就一个快速修复"看起来很明显
- 你已经试过多个修复
- 上一个修复没生效
- 你没有完全理解问题

**不要在以下情况跳过：**
- 问题看起来简单（简单的 bug 也有根因）
- 你在赶时间（赶工保证返工）
- 经理想要现在就修好（系统化比乱撞更快）

## 四个阶段

你必须完成每个阶段才能进入下一个。

### Phase 1：根因调查

**在尝试任何修复之前：**

1. **仔细读错误信息**
   - 不要跳过错误或警告
   - 它们通常包含确切的解决方案
   - 完整阅读堆栈跟踪
   - 记下行号、文件路径、错误码

2. **一致地复现**
   - 你能可靠地触发它吗？
   - 确切步骤是什么？
   - 每次都发生吗？
   - 如果不可复现 → 收集更多数据，不要猜

3. **检查最近的改动**
   - 什么改动可能导致了这个？
   - Git diff、最近的提交
   - 新依赖、配置改动
   - 环境差异

4. **在多组件系统中收集证据**

   **当系统有多个组件时（CI → build → signing，API → service → database）：**

   **在提出修复之前，添加诊断埋点：**
   ```
   对每个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置的传递
     - 检查每一层的状态

   运行一次，收集显示它在哪断裂的证据
   然后分析证据，定位失败的组件
   然后调查那个具体组件
   ```

   **示例（多层系统）：**
   ```bash
   # 第 1 层：工作流
   echo "=== Secrets available in workflow: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # 第 2 层：构建脚本
   echo "=== Env vars in build script: ==="
   env | grep IDENTITY || echo "IDENTITY not in environment"

   # 第 3 层：签名脚本
   echo "=== Keychain state: ==="
   security list-keychains
   security find-identity -v

   # 第 4 层：实际签名
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **这揭示：** 哪一层失败（secrets → workflow ✓，workflow → build ✗）

5. **追踪数据流**

   **当错误深在调用栈中时：**

   完整的反向追踪技术见本目录下的 `root-cause-tracing.md`。

   **快速版：**
   - 坏值源自哪里？
   - 谁用坏值调用了它？
   - 持续向上追踪，直到找到源头
   - 在源头修，而非在症状处

### Phase 2：模式分析

**在修复前找到模式：**

1. **找到可工作的示例**
   - 在同一代码库中找到类似的能工作的代码
   - 什么与坏掉的东西类似且能工作？

2. **与参考对比**
   - 如果在实现某个模式，完整地读参考实现
   - 不要略读——逐行读
   - 在应用前完全理解模式

3. **识别差异**
   - 能工作的和坏掉的之间有什么不同？
   - 列出每个差异，无论多小
   - 不要假设"那不重要"

4. **理解依赖**
   - 这还需要哪些其他组件？
   - 什么设置、配置、环境？
   - 它做了哪些假设？

### Phase 3：假设与测试

**科学方法：**

1. **形成单一假设**
   - 清楚陈述："我认为 X 是根因，因为 Y"
   - 写下来
   - 要具体，不要含糊

2. **最小化测试**
   - 做最小的可能改动来测试假设
   - 一次一个变量
   - 不要一次修多个东西

3. **在继续前验证**
   - 生效了吗？是 → Phase 4
   - 没生效？形成新假设
   - 不要在上面叠加更多修复

4. **当你不知道时**
   - 说"我不理解 X"
   - 不要假装知道
   - 寻求帮助
   - 多研究

### Phase 4：实现

**修根因，而非症状：**

1. **创建失败测试用例**
   - 最简单的可能复现
   - 尽可能用自动化测试
   - 没有框架就用一次性测试脚本
   - 修复前必须有
   - 用 `superpowers:test-driven-development` 技能编写规范的失败测试

2. **实现单一修复**
   - 应对识别出的根因
   - 一次一个改动
   - 不要"顺手"改进
   - 不要捆绑重构

3. **验证修复**
   - 测试现在通过了吗？
   - 没有破坏其他测试？
   - 问题真的解决了？

4. **如果修复没生效**
   - 停下
   - 数一数：你试过多少个修复？
   - 如果 < 3：回到 Phase 1，用新信息重新分析
   - **如果 ≥ 3：停下并质疑架构（下面的步骤 5）**
   - 不要在没有架构讨论的情况下尝试第 4 个修复

5. **如果 3+ 个修复失败：质疑架构**

   **表明架构问题的模式：**
   - 每个修复都揭示新的共享状态/耦合/别处的问题
   - 修复需要"大规模重构"才能实现
   - 每个修复都在别处制造新症状

   **停下并质疑根本：**
   - 这个模式根本合理吗？
   - 我们是否"出于惯性才坚持它"？
   - 我们该重构架构，还是继续修症状？

   **在尝试更多修复之前与你的搭档讨论**

   这不是失败的假设——这是错误的架构。

## 红旗 - 停下并遵循流程

如果你发现自己在想：
- "先快速修一下，以后再调查"
- "就改一下 X 试试看能不能工作"
- "加多个改动，跑测试"
- "跳过测试，我手动验证"
- "大概是 X，让我修一下"
- "我没有完全理解，但这可能管用"
- "模式说 X，但我会换个方式改造"
- "主要问题是这些：[没调查就列出修复]"
- 在追踪数据流之前就提出解决方案
- **"再试一次修复"（当已经试过 2+ 次时）**
- **每个修复都揭示别处的新问题**

**以上所有都意味着：停下。回到 Phase 1。**

**如果 3+ 个修复失败：** 质疑架构（见 Phase 4.5）

## 你搭档发出的"你做错了"信号

**留意这些纠偏：**
- "那不是没发生吗？" —— 你假设了却没验证
- "它会展示给我们……吗？" —— 你本该加证据收集
- "别猜了" —— 你在不理解的情况下提修复
- "把这个 ultra-think 一下" —— 质疑根本，不只是症状
- "我们卡住了？"（沮丧）—— 你的方法不奏效

**看到这些时：** 停下。回到 Phase 1。

## 常见合理化

| 借口 | 现实 |
|--------|---------|
| "问题很简单，不需要流程" | 简单问题也有根因。流程对简单 bug 很快。 |
| "紧急，没时间走流程" | 系统化调试比猜-检查-乱撞更快。 |
| "先试这个，再调查" | 第一个修复定下模式。从一开始就做对。 |
| "我等确认修复生效再写测试" | 没测试的修复立不住。先写测试证明它。 |
| "一次多个修复省时间" | 无法隔离什么生效。引入新 bug。 |
| "参考太长，我改造模式" | 部分理解保证出 bug。完整读它。 |
| "我看到问题了，让我修" | 看到症状 ≠ 理解根因。 |
| "再试一次修复"（2+ 次失败后） | 3+ 次失败 = 架构问题。质疑模式，不要再修。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|-------|---------------|------------------|
| **1. 根因** | 读错误、复现、检查改动、收集证据 | 理解是什么和为什么 |
| **2. 模式** | 找能工作的示例、对比 | 识别差异 |
| **3. 假设** | 形成理论、最小测试 | 假设确认或形成新假设 |
| **4. 实现** | 创建测试、修复、验证 | bug 解决，测试通过 |

## 当流程揭示"没有根因"时

如果系统化调查揭示问题确实是环境性的、依赖时序的或外部的：

1. 你已经完成了流程
2. 记录你调查了什么
3. 实现合适的处理（重试、超时、错误信息）
4. 添加监控/日志以便日后调查

**但是：** 95% 的"没有根因"案例是不完整的调查。

## 支持技术

## 真实世界影响

来自调试会话：
- 系统化方法：15-30 分钟修复
- 随机修复方法：2-3 小时乱撞
- 一次修复成功率：95% vs 40%
- 引入新 bug：接近零 vs 常见
